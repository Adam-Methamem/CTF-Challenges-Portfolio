Matryoshka.elf

ELF binary forensics  ·  4-layer chained XOR extraction  ·  password-protected ZIP

A valid Linux binary that runs clean and exits 0. Every standard tool sees nothing unusual. Four layers of data are hidden across spec-legal-but-tool-invisible locations in the ELF structure, each encrypted with a key recovered from the previous layer. The flag lives inside a XOR-obfuscated, password-protected ZIP at the tail of the file.

The four layers at a glance

LAYER 1SHT tail — the key
offset 0x11d0  ·  32 bytes after SHT end
No tool reads past the last Section Header Table entry. First 16 bytes = XOR key for Layer 2.
LAYER 2PT_NOTE desc — encrypted
offset 0x1010  ·  note description field
Unknown note type forces readelf to hex-dump instead of decode. XOR with L1 key → L2 key.
LAYER 3.gs section — encrypted
offset 0x1050  ·  sh_type=0xDEAD
Invalid sh_type causes readelf to skip section content entirely. XOR with L2 key → ZIP password.
LAYER 4XOR'd ZIP — the flag
offset 0x11f0  ·  ELF tail
Entire ZIP is XOR'd so PK magic disappears. First byte = XOR key. Password from Layer 3.

Step 0 — first look

$ file matryoshka.elf
matryoshka.elf: ELF 64-bit LSB executable, x86-64, statically linked

$ ./matryoshka.elf ; echo $?
0

$ strings matryoshka.elf
(only garbage — no readable layer content, no flag, no filenames)

Runs fine. Exits 0. strings shows nothing useful. The binary is 4756 bytes — far too small to contain real code, which tells you everything interesting is hidden data.

Step 1 — inspect sections

$ readelf -S matryoshka.elf
  [Nr] Name   Type              Offset   Size
  [ 1] .tx    PROGBITS          0x1000   0x00c
  [ 2] .nt    NOTE              0x1010   0x034
  [ 3] .gs    0000dead: <unkn   0x1050   0x020
  [ 4] .ss    STRTAB            0x1070   0x011

Section [3] .gs has sh_type = 0x0000DEAD — not a valid ELF section type. readelf prints <unkn> and silently skips the section content. This is the first anomaly. Note the offset 0x1050 and size 0x20 (32 bytes).

Layer 3 — dump .gs section manually

readelf won't show the content — but you have the offset. Go directly to the bytes.

$ python3 -c "d=open('matryoshka.elf','rb').read(); print(d[0x1050:0x1050+32].hex())"
33f7f943c20aed62e036f7415ef3eb6527f5fa37f16eb209d34fa8712ec0853a

This is Layer 3 data — XOR-encrypted with Layer 2's key. Save it. You need to decrypt Layer 2 first.

Step 2 — inspect the PT_NOTE segment

$ readelf -n matryoshka.elf
Displaying notes found in: .nt
  Owner       Data size    Description
  ����        0x00000020   Unknown note type: (0xdeadbeef)

                    10 42 68 70 8f 82 36 5c a1 d6 2b e6 e8 da 81 09

The vendor name is non-ASCII binary bytes — unreadable. Note type 0xdeadbeef is not recognised, so readelf dumps the description as raw hex instead of decoding it. Those 32 hex bytes are Layer 2 data, encrypted with the Layer 1 key.

Layer 1 — bytes after the Section Header Table

Why this works: The ELF spec ends at offset e_shoff + e_shnum × 64 — the last byte of the last section header entry. Nothing in the spec forbids appending data after that point. No tool — readelf, objdump, nm, file — reads past it. The bytes are completely invisible unless you compute the offset manually. 
$ python3 -c "
import struct
d = open('matryoshka.elf','rb').read()
e_shoff = struct.unpack_from('<Q', d, 0x28)[0]
e_shnum = struct.unpack_from('<H', d, 0x3C)[0]
sht_end = e_shoff + e_shnum * 64
print('SHT ends at:', hex(sht_end))
print(d[sht_end : sht_end+32].hex())
"
SHT ends at: 0x11d0
a37f2cd1884e63b519f05a3ce7921d6b deadbeefcafebabe1337c0defaceff00

The first 16 bytes — a37f2cd1884e63b519f05a3ce7921d6b — are Layer 1: the XOR key for Layer 2. The remaining 16 bytes are noise padding.

Decrypting the chain

$ python3 -c "
def xor(data, key):
    return bytes(b ^ key[i % len(key)] for i,b in enumerate(data))

# Layer 1 key (first 16 bytes after SHT)
L1_KEY  = bytes.fromhex('a37f2cd1884e63b519f05a3ce7921d6b')

# Layer 2 data (hex dump from readelf -n)
L2_DATA = bytes.fromhex('febba6e67920d1bccabff24dc9529851104268708f82365ca1d62be6e8da8109')
L2_KEY  = xor(L2_DATA, L1_KEY)[:16]
print('L2_KEY:', L2_KEY.hex())

# Layer 3 data (hex dump from .gs section)
L3_DATA  = bytes.fromhex('33f7f943c20aed62e036f7415ef3eb6527f5fa37f16eb209d34fa8712ec0853a')
ZIP_PASS = xor(L3_DATA, L2_KEY)[:19]
print('ZIP pass:', ZIP_PASS.decode())
"
L2_KEY: 5dc48a37f16eb209d34fa8712ec0853a
ZIP pass: n3st3d_k3y_0p3n_z1p

Layer 4 — finding and extracting the ZIP

The ZIP is not stored raw — it is XOR'd with a single byte key so the PK\x03\x04 magic disappears and no tool recognises it. The first byte of the blob at the tail is the XOR key itself.

$ python3 -c "
import struct, zipfile, io

def xor(data, key):
    return bytes(b ^ key[i % len(key)] for i,b in enumerate(data))

d = open('matryoshka.elf','rb').read()

# ZIP blob starts at SHT_end + 32 (Layer 1 size)
e_shoff  = struct.unpack_from('<Q', d, 0x28)[0]
e_shnum  = struct.unpack_from('<H', d, 0x3C)[0]
sht_end  = e_shoff + e_shnum * 64
zip_blob = d[sht_end + 32:]

# First byte = XOR key, rest = XOR'd ZIP
xor_key = zip_blob[0]
raw_zip  = bytes([b ^ xor_key for b in zip_blob[1:]])
print('XOR key:', hex(xor_key))
print('ZIP magic:', raw_zip[:4].hex())   # must be 504b0304

# Extract with password from Layer 3
zf   = zipfile.ZipFile(io.BytesIO(raw_zip))
flag = zf.read('d', pwd=b'n3st3d_k3y_0p3n_z1p').decode()
print(flag)
"

Full solution chain

LAYER 1 → bytes past SHT end → first 16 bytes = L1_KEY
LAYER 2 → XOR(note_desc, L1_KEY) → first 16 bytes = L2_KEY
LAYER 3 → XOR(.gs data, L2_KEY) → first 19 bytes = ZIP password
LAYER 4 → XOR(zip_blob[1:], zip_blob[0]) → raw ZIP → flag.txt

Why each layer is invisible to tools

LayerLocationWhy tools miss it
1 — SHT tail After offset 0x11d0 ELF spec ends at last SHT entry. readelf, objdump, nm — none read past it.
2 — PT_NOTE Note desc at 0x1010 note_type 0xdeadbeef is unknown — readelf hex-dumps instead of decoding. Vendor is non-ASCII binary bytes.
3 — .gs section Section at 0x1050 sh_type 0xDEAD is invalid — readelf prints <unknown> and skips section content entirely.
4 — ZIP Tail at 0x11f0 Entire ZIP XOR'd with 0x42 — PK magic gone. file reports ELF. binwalk finds nothing. strings shows garbage.

The core lesson: tools lie by omission.
Every hiding spot in this file is spec-valid. The OS runs the binary without complaint.
Solving it requires knowing what the tools choose not to show — and going directly to the bytes.
